## O10: Semantic Gap Detection (3 hours)

> **TODO**: rellenar A/B/C antes de aprobar Phase 4.

### Purpose
Automatically identify topics where claims exist but lack decisive guidance or clear action steps.

### Implementation

1. **Create `scripts/gap-detector.py`**
   - Input: VAULT_PATH
   - Output: `meta/semantic-gaps.md`
   - Logic:
     - For each MOC, collect all atoms under each subtopic
     - Score each atom's actionability (1-10):
       - Present specific numbers/conditions? +3
       - Contain decision tree or "if-then"? +3
       - Mention "always/never" or clear boundary? +2
       - Provide example? +2
     - Average score per subtopic
     - If average < 6: flag as "gap - low actionability"

2. **Actionability scoring**
   - Read atom's body and claim
   - Presence checks:
     - "if", "only if", "when": +2
     - Numbers, percentages, thresholds: +3
     - "Do X" (imperative): +2
     - "See also" links to related atoms: +1
   - Cap score at 10

3. **Gap classification**
   ```
   **Low Actionability Gap** (score 0-5):
   - [[topic]]: {{N}} atoms but average actionability {{score}}/10
   - Problem: advice is vague, lacks decision criteria
   - Suggestion: ingest case studies or concrete examples
   
   **Coverage Gap** ({{N}} atoms total):
   - [[topic]]: has atoms but missing subtopic
   - Problem: sub-question unanswered
   - Suggestion: create atom for {{missing-subtopic}}
   ```

4. **Gap report format**
   ```markdown
   # Semantic Gaps — {{YYYY-MM-DD}}
   
   **Summary**: {{N}} identified gaps across {{M}} topics
   
   ## Low Actionability Gaps (advice too vague)
   - [[pricing]]: average actionability 4.2/10
     - Atoms: [[atom-1]], [[atom-2]], [[atom-3]]
     - Problem: pricing strategies described but no thresholds
     - Suggestion: ingest PriceLabs tutorial or case studies
   
   ## Coverage Gaps (missing subtopics)
   - [[market-selection]]: Missing "negative filter" subtopic
     - Related atom: [[market-selection--positive-filters]]
     - Suggestion: Create atom for negative filters
   
   ## Generated by
   `scripts/gap-detector.py` on {{timestamp}}
   ```

5. **Integration with agent**
   - Run gap-detector as part of `scripts/vault-agent.py`
   - Include gaps in health report

6. **Git commit**
   ```bash
   git commit -m "O10: Semantic gap detection (gap-detector.py)"
   ```

## O11: Backlink Generation (4 hours)

> **TODO**: rellenar A/B/C antes de aprobar Phase 4.

### Purpose
Automatically generate `meta/backlinks.md` showing influence map (what cites what).

### Implementation

1. **Create `scripts/backlink-gen.py`**
   - Input: VAULT_PATH
   - Output: `meta/backlinks.md`
   - Scans all atom bodies for `[[links]]`
   - Groups by: inbound (cites me), outbound (I cite)
   - Categorizes by link strength:
     - Direct citation in body: strength=HIGH
     - Mentioned in "See also": strength=MEDIUM
     - Mentioned in `conflicts_with`: strength=HIGH (conflict)

2. **Inbound link aggregation**
   - For each atom, find all other files that cite it
   - Format:
     ```
     ## [[atom-name]]
     
     **Inbound ({{N}} cites)**:
     - [[query-1]] → high (direct answer)
     - [[atom-2]] → medium (see also)
     - [[atom-3]] → high (conflict)
     
     **Outbound ({{M}} cites)**:
     - → [[atom-4]] (related)
     - → [[source-1]] (evidence)
     ```

3. **Link strength visualization**
   - HIGH: `**[[link]]**` (bold) — core dependency or conflict
   - MEDIUM: `[[link]]` — reference or suggestion
   - LOW: `*[[link]]*` (italic) — passing mention

4. **Hub detection**
   - Atoms with 5+ inbound links = HUB (likely important)
   - Atoms with 0 inbound links = ORPHAN (should link or delete)
   - Atoms with 1 inbound link = LEAF (specialized, niche)
   - Output summary:
     ```
     ## Link Statistics
     - Hubs (5+ inbound): {{N}} atoms
     - Leaves (1 inbound): {{N}} atoms
     - Orphans (0 inbound): {{N}} atoms
     - Average inbound per atom: {{X}}
     ```

5. **Automated maintenance**
   - Call `backlink-gen.py` weekly via vault-agent.py
   - Detect new orphans created by recent ingests
   - Flag orphans in agent report for user review

6. **Visualization enhancement (optional)**
   - Obsidian graph view can color nodes by inbound link count
   - Bright nodes = hubs (important); dim = leaves
   - Integrate via frontmatter: `hub_level: {{score}}`

7. **Git commit**
   ```bash
   git commit -m "O11: Backlink generation (backlink-gen.py + meta/backlinks.md)"
   ```

### Verification Checklist
- [ ] `scripts/backlink-gen.py` created and executable
- [ ] Scans all atom bodies for `[[links]]`
- [ ] Categorizes links by strength (HIGH/MEDIUM/LOW)
- [ ] Aggregates inbound/outbound per atom
- [ ] `meta/backlinks.md` generated with proper format

## O12: RAG Fallback Integration (6 hours)

> **TODO**: rellenar A/B/C antes de aprobar Phase 4.

### Purpose
Implement hybrid search: primary atom-based, fallback to embedding-based RAG if needed.

### Implementation

1. **Choose vector DB backend** (select one)
   - Option A: **qmd** — lightweight, local, BM25 + vector, MCP server available
   - Option B: **Ollama + simple similarity** — local LLM + cosine similarity, ~50 lines Python
   - Option C: **OpenAI embeddings** — requires API key, cloud-based
   - **Recommendation**: qmd (ready-made, no extra setup)

2. **Install and configure qmd** (if Option A)
   ```bash
   # On macOS
   brew install qmd  # hypothetical; may need manual install
   
   # Index vault sources
   qmd index $VAULT_PATH/sources/
   ```

3. **Create `scripts/hybrid-search.py`**
   - Layer 1 (primary): atom search (existing logic from CLAUDE.md §4.2 steps 3-5)
   - If Layer 1 returns 0 atoms:
     - Trigger Layer 2: embedding search over sources
     - Query: `qmd search --query "user question" --top 5 $VAULT_PATH/sources/`
     - Parse results: extract top 2-3 sources
     - Read sources with offset/limit around relevant timestamps
     - Synthesize answer from raw sources
     - Mark response: "Based on source fragments (lower confidence than atoms)"

4. **Response contract for RAG fallback**
   ```markdown
   ⚠️ **Note**: No direct atoms found. Answer synthesized from source fragments.
   
   {{Answer synthesized from sources}}
   
   **Confidence**: Medium (lower than atom-based responses)
   **Sources**: [[source-A#t=MM:SS]], [[source-B#t=MM:SS]]
   
   **Suggestion**: Consider creating atoms for {{missing-topic}} to improve coverage.
   ```

5. **Integration with query pipeline (§4.2)**
   - After step 6 (shortlist atoms):
   - If shortlist.length < 2:
     - Log: "Shortlist sparse, triggering RAG fallback"
     - Call `hybrid-search.py --fallback --query "{{user_question}}"`
     - Merge Layer 2 results into response
     - Flag as fallback in response

6. **Cost tracking**
   - Layer 1 query cost: ~8k tokens
   - Layer 2 fallback: ~15k tokens
   - Track in `meta/query-cache-stats.md`:
     ```
     ## Fallback Usage
     - Fallback queries: {{N}} / {{total}} ({{percent}}%)
     - Topics requiring fallback most: {{topic-1}}, {{topic-2}}
     - Average Layer 2 cost: {{15k}} tokens
     ```

7. **Graceful degradation**
   - If vector search also fails: return "I don't have relevant information"
   - Never hallucinate; always cite or say "unknown"

8. **Test Layer 2 activation**
   - Create query with no atom coverage
   - Verify: fallback triggers
   - Verify: response includes caveat
   - Verify: sources cited correctly with timestamps

9. **Git commit**
   ```bash
   git commit -m "O12: RAG fallback (hybrid-search.py + qmd integration)"
   ```

- [ ] qmd installed and indexed (if Option A) or Ollama configured (if Option B)
- [ ] `scripts/hybrid-search.py` created and executable
- [ ] Layer 1 (atom search) still works as before
- [ ] Layer 2 (RAG fallback) triggers when atom shortlist < 2
